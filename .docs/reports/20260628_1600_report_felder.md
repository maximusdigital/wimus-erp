## 2. Abweichungen von der Spec (begründet)

- **Kontaktmodell NICHT neu gebaut, sondern erweitert.** Real ist `kontakte` Single-Table-polymorph (`kontakt_typ` person|firma + Rollen-Flags `ist_mieter/…` + `organisation_id` 1:n + `firma_id` Self-FK); `organisationen` (Migration 012) existiert bereits als externer Geschäftspartner-Stamm. Statt getrennter neuer `personen`/`organisationen`-Tabellen (Spec §1) → **„Person" = `kontakte(kontakt_typ='person')`, „Organisation" = `organisationen`**. Das vermeidet einen riskanten Umbau der laufenden kontakte-UI/-API/-Validierung/-Relations (CLAUDE.md: „Vorhandenes erweitern, nicht neu daneben"). Ergänzt wurden nur die **fehlenden** Stücke: n:m-Typen + n:m person↔organisation.
- **Variante C gewählt** (typisierte EAV-Wert-Spalten) — Begründung s. §3.
- **App läuft real auf `wimus`** (nicht `public` wie in CLAUDE.md/Stand-Sektion behauptet): `lib/supabase/server.ts` Default `db.schema=wimus`, kontakte-Seite/-API chainen explizit `.schema("wimus")`. Hinweis für Konzept-Claude: CLAUDE.md-Cutover-Stand ist veraltet.
- **Feldtypen-Mapping:** Spec-Typ `janein` wird auf Speicherspalte `wert_bool` abgebildet (DB-feldtyp `janein` additiv erlaubt); `auswahl` speichert den stabilen `opt_key` in `wert_text`. Reine Servicelogik, für Konsumenten transparent.

## 3. Entscheidungen (im Auftrag offen gelassen)

**Speicher-Variante: C (typisierte EAV-Wert-Spalten).** Begründung:
- Unsere Größenordnung (einige tausend Kontakte, wenige Felder je Entität) → Joins unkritisch; die Fachwelt-Warnung gilt v.a. bei Millionen Zeilen / sehr vielen sparse Feldern.
- C liefert **physische, typisierte, indizierbare** Wert-Spalten (sauber filter-/sortierbar je Feld, partielle B-Tree-Indizes) und vermeidet das „alles-als-Text"-Antipattern — genau das, was die ungenutzte 002-`custom_field_werte` (nur `wert text`) war.
- **0006-Andockung sauberer:** Der 0006-Query-Builder kennt nur **flache** Spalten. Variante B (JSONB) hätte Casting/GIN-Logik in den Query-Builder gedrückt (zweite Filtersystem-Komplexität). C dockt stattdessen über einen **id-Prefilter** an: `customFieldIds()` löst einen Custom-Field-Filter zu `bezug_id`-Mengen auf → der List-Route hängt `.in("id", ids)` an die bestehende Entitäts-Query. Kein Eingriff in den Kern-Query-Builder, kein zweites Filtersystem.
- B (JSONB-Hybrid) bleibt hinter derselben Service-Schnittstelle migrierbar, falls die reale Multi-Feld-Filterlast später messbar bremst.

**`lieferanten` ↔ Organisationstyp „Lieferant": bewusst zweigleisig, je eine Wahrheit pro Zweck.**
- `wimus.lieferanten` (Migration 010) = **Buchungs-Kreditorenstamm** (FK `firma_id` INNEN, `standard_konto`, `standard_gewerk`, `alias[]` für OCR-Fuzzy via `lib/fibu/lieferant-match.ts`, referenziert von `belege.lieferant_id`). Buchungs-/FiBu-Entität.
- Kontakt-/Org-Typ `lieferant` = **Geschäftspartner-Klassifikation** (CRM/Kontaktsicht). 
- Nicht zusammengelegt (unterschiedliche Pflichtfelder/Zweck). Optionaler späterer Link `lieferanten.organisation_id` als Roadmap notiert — nicht in Stufe 1.

**Welche Entitäten Custom Fields:** person, organisation, vorgang, objekt, einheit (alle in der UI wählbar; bezug_typ-CHECK deckt sie ab).

## 4. Offen / Rückfragen (geparkt)

1. **0006-List-Route-Verdrahtung (Stufe 2):** `defsToFilterFields` + `customFieldIds` (id-Prefilter) sind gebaut **und getestet**, aber noch nicht in eine konkrete Listen-/Filter-Route eingehängt (die kontakte-Liste hat aktuell gar keine filter-bar-Route). Sobald die 0006-Filterleisten-Route je Entität steht, dort die dynamischen Felder zuschalten. **0006-Spec-Ergänzung „dynamische Felder" terminieren** (Konzept-Claude).
2. **Custom-Field-Werte in der Entitäts-Detailansicht (Stufe 2):** `getWerte`/`setWert` sind fertig; das Rendern/Editieren der Custom Fields in den `kontakte/[id]`- bzw. organisationen-Detailseiten + die n:m-Typen-/Ansprechpartner-Zuweisungs-UI (`setZuordnungen`/`person_organisation`) sind noch nicht verdrahtet (Service-API steht, UI parken).
3. **Organisationen-CRUD-UI:** `organisationen` hat noch keine eigene Verwaltungs-UI (nur DB + Kontakt-Verknüpfung). Falls gewünscht: eigener `/kontakte`-Tab oder `/organisationen`-Route.
4. **System-Typen vs. `ist_*`-Flags:** Person-System-Typen spiegeln die bestehenden `kontakte.ist_*`-Booleans (von `lib/fibu` + `api/kontakte ?rolle=` genutzt). Beide existieren parallel; eine spätere Vereinheitlichung (Flags → Typ-Zuordnung) wäre ein separater Migrationsschritt — bewusst nicht erzwungen.
5. **SQL-Einspielung:** Migration 027 ist **nicht** eingespielt (DB-Port zu). Anwenden über Supabase SQL-Editor; danach Seed der System-Typen prüfen (`SELECT gilt_fuer, count(*) FROM wimus.kontakt_typen WHERE geschuetzt GROUP BY 1`).
