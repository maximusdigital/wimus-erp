## Was zu bauen ist

### 1. Migration `supabase/migrations/030_tags.sql` (idempotent)
> Nächste freie Nummer ist 030 (029_org_phase_b0.sql existiert bereits, NICHT anfassen).

Zwei Tabellen im Schema `wimus`:

**tags**
- id uuid PK default gen_random_uuid()
- mandant_id uuid NOT NULL REFERENCES wimus.mandanten(id) ON DELETE CASCADE
- label text NOT NULL
- farbe text NULL  (UI-Chip-Farbe, optional)
- aktiv boolean NOT NULL default true
- created_at / updated_at timestamptz NOT NULL default now()
- **UNIQUE INDEX ux_tags_mandant_label ON wimus.tags (mandant_id, lower(label))**
  (case-insensitive eindeutig je Mandant — „Wasserschaden" == „wasserschaden")

**tag_zuordnung**
- id uuid PK default gen_random_uuid()
- tag_id uuid NOT NULL REFERENCES wimus.tags(id) ON DELETE CASCADE
- bezug_typ text NOT NULL   (FREIER String, KEINE CHECK-Whitelist — jede Tabelle taggbar)
- bezug_id uuid NOT NULL    (polymorph, KEIN FK)
- created_at timestamptz NOT NULL default now()
- **UNIQUE (tag_id, bezug_typ, bezug_id)**
- INDEX ix_tagzuordnung_bezug ON wimus.tag_zuordnung (bezug_typ, bezug_id)
- (tag_id-Index entsteht über den FK ohnehin)
- bewusst KEIN updated_at (Zuordnungen werden erstellt/gelöscht, nicht editiert)

**RLS (Muster wie bestehende Kerntabellen / custom_field_werte):**
- tags: `mandant_isolation` über eigene mandant_id (USING + WITH CHECK
  `mandant_id IN (SELECT wimus.user_mandanten())`).
- tag_zuordnung: `mandant_isolation` über das Eltern-Tag
  (`EXISTS (SELECT 1 FROM wimus.tags t WHERE t.id = tag_zuordnung.tag_id
   AND t.mandant_id IN (SELECT wimus.user_mandanten()))`).

**updated_at-Trigger** auf tags (greift automatisch über die bestehende DO-Schleife in 002, die alle
Tabellen mit updated_at-Spalte abdeckt — prüfen, ob sie auch nachträglich angelegte Tabellen erfasst;
falls nicht, Trigger explizit setzen). tag_zuordnung braucht keinen (kein updated_at).

**GRANTS** analog Bestand (authenticated, service_role, anon usage) — Muster aus 002/004 übernehmen.

**KEIN Seed** — Tags entstehen zur Laufzeit durch Nutzer.

Idempotent durchgängig: CREATE TABLE IF NOT EXISTS, CREATE INDEX IF NOT EXISTS, DROP POLICY IF
EXISTS + CREATE POLICY. Sequenziell, kein force.
